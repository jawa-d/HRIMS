**تاريخ التقرير:** 8 مارس 2026

## 1. الملخص التنفيذي
هذا التقرير يوضح آلية عمل النظام بشكل شامل، مع تقييم مستوى امتياز النظام وجاهزيته للتشغيل الفعلي.  
النظام مبني على واجهة ويب حديثة (HTML/CSS/Vanilla JS Modules) ومربوط بخدمات Firebase (Authentication + Firestore)، ويطبق صلاحيات حسب الدور الوظيفي مع تدفق عمل واضح للموارد البشرية والرواتب والحسابات.

**الخلاصة:**  
النظام قوي من ناحية الهيكل والوظائف الأساسية، وجاهزيته التشغيلية **مرتفعة**، خصوصاً بعد تثبيت آخر تحسينات الرواتب والحسابات.

## 2. نظرة معمارية على النظام
## 2.1 طبقات النظام
- **طبقة العرض:** صفحات HTML + ملفات CSS مخصصة لكل وحدة.
- **طبقة التحكم:** `Page Controllers` لكل صفحة (منطق الواجهة والأحداث).
- **طبقة الخدمات:** `Services` للوصول للبيانات (Firestore) والعمليات المشتركة.
- **طبقة الحماية:** `Aman/guard.js` + صلاحيات مركزية من `app.config.js`.

## 2.2 الوحدات التشغيلية الرئيسية
- إدارة الموظفين (`employees`)
- الحضور والانصراف (`attendance`)
- الإجازات (`leaves`)
- الرواتب (`payroll`)
- المحاسبة العامة (`accounting`)
- تدفق النقد (`accounting_flow`)
- الإدارة المحاسبية المتقدمة (`accounting_admin`)
- التقارير (`reports`)
- الإشعارات (`notifications`)
- الإعدادات والصلاحيات (`settings`)

## 2.3 الأدوار والصلاحيات
الأدوار المعتمدة:
- `super_admin`
- `hr_admin`
- `manager`
- `employee`

الصلاحيات مطبقة على مستويين:
- منع الوصول للصفحة غير المصرح بها (`enforceAuth`)
- تحكم دقيق بالأفعال داخل الصفحة (`canDo` / `ACTION_PERMISSIONS`)

## 3. آلية العمل التشغيلية
## 3.1 دورة العمل العامة
1. المستخدم يسجل الدخول عبر Firebase Auth.
2. النظام يحدد الدور ويحمّل الواجهة المناسبة.
3. كل صفحة تنفذ Controller خاص بها.
4. Controller يجلب البيانات من Services ثم يعرضها.
5. أي حفظ/تعديل يتم كتابته إلى Firestore مع تحديثات الحالة.
6. يتم تسجيل أحداث الاستخدام (Telemetry) وإظهار إشعارات عند الحاجة.

## 3.2 آلية عمل الرواتب (Payroll)
- تحميل الموظفين الفعالين فقط.
- تحميل رواتب الشهر المحدد.
- إدخال/تعديل: `Base`, `Allowances`, `Deductions`.
- احتساب صافي الراتب تلقائياً:  
  `Net = Base + Allowances - Deductions`
- حالات الشهر: `Draft` / `In Progress` / `Approved`.
- عمليات أساسية: حفظ مسودة، اعتماد الشهر، تصدير CSV/PDF.
- **ميزة مضافة حديثاً:** ترتيب الموظفين بالسحب والإفلات داخل الجدول مع حفظ الترتيب لكل شهر محلياً.

## 3.3 آلية عمل الحسابات (Accounting)
- لوحة مالية تعرض:
  - الافتتاحي
  - الداخل
  - الخارج
  - المصروف
  - الصافي
  - مؤشرات الالتزامات والسلف
- إدخالات سريعة (قبض/صرف) وتحرير القيود.
- تصدير بيانات التحليل إلى Excel.
- ربط مباشر بصفحة التدفق النقدي وصفحة الصندوق.

## 3.4 آلية عمل المحاسبة الإدارية (Accounting Admin)
- إدارة دليل الحسابات (Chart of Accounts).
- إدارة الالتزامات: عهدة / سلفة / ذمم مدينة / ذمم دائنة.
- دورة السلفة: `requested -> approved -> disbursed -> closed` (مع رفض عند الحاجة).
- قفل وإعادة فتح الشهر/السنة المحاسبية.
- تصدير بيانات الالتزامات والحسابات (Excel/PDF).

## 4. ما يميز النظام (نقاط الامتياز)
- بنية واضحة وقابلة للتوسعة.
- فصل جيد بين واجهة الاستخدام ومنطق الأعمال والخدمات.
- صلاحيات Role-Based دقيقة وليست سطحية.
- دعم عربي/إنجليزي + RTL/LTR.
- واجهات تشغيلية تغطي الموارد البشرية والمحاسبة ضمن نظام واحد.
- أدوات تصدير متعددة (CSV/Excel/PDF) لسهولة العمل الإداري.
- وجود وحدات مساندة مهمة: مركز الإشعارات، الأمان، صحة النظام، إدارة الصفحات.

## 7. توصيات قبل الإطلاق الرسمي
1. تنفيذ Smoke Test نهائي على صفحات: `payroll`, `accounting`, `accounting_admin`, `accounting_flow`.
2. اختبار صلاحيات كل دور فعلياً بحسابات حقيقية (خاصة صلاحيات الحذف/الاعتماد/القفل).
3. مراجعة Firestore Rules على بيئة الإنتاج بعد أي تعديل.
4. تفعيل نسخ احتياطي دوري لبيانات Firestore.
5. اعتماد إجراء تشغيلي ثابت لنهاية كل شهر (رواتب + إغلاق محاسبي + تقارير).
